## 2026-03-25 late-night follow-up — token lifecycle hardening after red-team post-patch audit

Follow-up session focused on the remaining token-lifecycle holes found by the hostile audit after the earlier FM-4/FM-5/FM-11 patches. The goal of this pass was to remove the newly confirmed production risks rather than just documenting them.

### Issues fixed in this pass

#### 1. Rehydrate failures no longer delete the global session and kill the poller

Previously, `ProlificPollerDurableObject.rehydrateSessionFromGlobal()` treated a failed refresh validation as a terminal invalid session:
- deleted the global session record
- cleared DO metadata
- caused `alarm()` to return without scheduling the next run

That meant a single transient refresh failure during DO rehydration could silently stop monitoring forever.

This pass changed the behavior so that failed rehydration validation:
- clears only the DO-local auth cache
- keeps the global user session record intact
- keeps DO metadata intact
- logs `poller_session_rehydrate_retry`
- causes `alarm()` to reschedule with the failure retry base interval instead of dying

This restores repeated validation attempts and allows the existing FM-4 failure counter / operator alert path to accumulate across rehydrate cycles.

#### 2. Refresh responses now preserve the previous refresh token when the server omits a replacement

`AuthClient.refreshTokens()` previously replaced the stored session with whatever `mapTokenResponse()` returned. If the OAuth server returned a successful refresh response without a `refresh_token` field, the stored refresh token became `undefined`, destroying future refresh capability.

This pass changed `mapTokenResponse()` to accept a fallback refresh token and `refreshTokens()` now preserves the old refresh token when the response omits a new one.

#### 3. Refresh responses now reject OAuth error payloads even when HTTP status is 200

`refreshTokens()` previously trusted any `200 OK` JSON body and relied on `mapTokenResponse()` to throw later when `access_token` was missing. That obscured the actual OAuth error reason.

This pass now:
- parses the refresh response body once
- rejects `{"error": ...}` payloads even on `200 OK`
- includes the upstream OAuth error / description in the thrown message
- rejects invalid non-JSON success bodies explicitly

#### 4. Durable Object `/trigger` failures are now non-fatal for `/vm/token-inject`

The Worker previously treated DO trigger failures the same as session bootstrap failures and returned `502 inject_partial` even when:
- global KV write succeeded
- the DO already had a valid session

### 00:11Z — traced and fixed `/vm/token-inject` 1101 production failure

#### Root cause trace

The `1101` was not a generic mystery runtime crash. The failing route was:
- `POST /vm/token-inject`
- `handleVmTokenInject(...)`
- `StateStore.putUserSession(record)`
- fallback write into `AUTH_CACHE` KV because `USER_SESSION_STORE` was **defined in code** but **not bound in `config/wrangler.toml`**

That meant production was still persisting linked user-session records through KV on the VM inject path. Once production KV writes hit the quota wall, `/vm/token-inject` crashed before it could return a structured JSON error, yielding Worker `1101`.

The earlier clue about Cloudflare API code `10048` was therefore directionally correct, but the sharper route-level cause was:
- **missing `USER_SESSION_STORE` binding in live Wrangler config left `/vm/token-inject` on the KV write path**

#### Patch applied

Repo/runtime changes:
- Restored `USER_SESSION_STORE` Durable Object bindings in:
  - root `[durable_objects]`
  - `[env.staging.durable_objects]`
  - `[env.production.durable_objects]`
- Kept `UserSessionDurableObject` as the authoritative global user-session store for new writes.
- Hardened `StateStore` so when `USER_SESSION_STORE` is present it still:
  - falls back to legacy KV user-session records on `getUserSession(...)`
  - lazily migrates those legacy KV records into the DO on read/list
  - merges DO records + legacy KV records in `listUserSessionRecords()`
  - best-effort deletes legacy KV copies on delete
- Updated docs to state that user sessions now persist primarily in `USER_SESSION_STORE` rather than only in KV.

Important deployment nuance discovered during the fix:
- attempting to add a fresh Wrangler migration for `UserSessionDurableObject` failed with Cloudflare API code `10074`
- message: `Cannot apply new-sqlite-class migration to class 'UserSessionDurableObject' that is already depended on by existing Durable Objects`
- conclusion: the class already existed in the live script history, so production only required the **binding restoration**, not a replayed new-sqlite migration
